핵심 문제 진단

TTS는 tts_script ("칠만천칠백구십삼 달러" — 발음 13자)로 음성을 생성하지만,
자막 타이밍은 display_script ("$71,793" — 표기 7자)의 단어 수로 비례 배분합니다.
→ "$71,793"는 1단어이지만 음성에서 3초 이상 차지 → 뒤 자막이 앞당겨져 싱크 깨짐

1
아키텍처
tts_script로 타이밍 계산
TTS 엔지니어 관점
타이밍 계산 시 display_script 대신 tts_script의 단어 수를 사용. 음성이 실제로 읽는 텍스트 기준으로 비례 배분하므로 근본적으로 정확. 자막 렌더링만 display_script 청크로 매핑.
tts_script → chunk → 타이밍 계산
display_script → chunk → 자막 렌더링
타이밍 + 자막 = 최종 ASS
+ 근본 원인 해결, 추가 API 불필요
- 두 스크립트의 청크 수 불일치 가능

API 활용
Google TTS Timepoint API
클라우드 API 전문가 관점
Google Cloud TTS의 enableTimePointing 옵션으로 각 단어/음절별 정확한 시작 시간을 API 응답에서 수신. silence detection 없이 정확한 워드레벨 타이밍 확보.
TTS API → audio + timepoints[]
timepoints로 청크 경계 결정
→ 정확한 자막 타이밍
+ 가장 정확한 워드레벨 타이밍
- v1beta1 전용, Chirp3-HD 지원 미확인

오디오 분석
Whisper 강제 정렬
음성인식 엔지니어 관점
생성된 TTS 오디오에 OpenAI Whisper를 실행하여 word-level timestamp를 추출. tts_script의 각 단어가 오디오 어디에 위치하는지 정확히 파악 후 자막 매핑.
TTS audio → Whisper STT
→ word timestamps 추출
→ 문장 경계에서 자막 분할
+ 언어 무관 정확한 정렬

문장 단위 개별 TTS
프로덕션 엔지니어 관점
전체 스크립트를 한 번에 TTS하지 않고, 문장별로 개별 TTS 호출. 각 문장의 정확한 오디오 길이를 측정 후 연결. 자막 타이밍 = 각 문장 오디오 길이의 누적합.
문장1 → TTS → 3.2s
문장2 → TTS → 4.1s
concat → 자막 타이밍 확정
+ 문장 단위 100% 정확한 타이밍
- API 호출 수 증가, 문장 간 이음새 부자연스러울 수 있음

알고리즘
음절 수 기반 비례 배분
언어학자 관점
단어 수 대신 tts_script의 음절 수로 비례 배분. 한국어는 음절 = 글자 수이므로 정확한 발화 시간 추정 가능. "칠만천칠백구십삼"=8음절 vs "$71,793"=1단어 → 음절이 더 정확.
tts_script 음절 카운트
→ 비례 배분 + silence snap
→ display_script 자막에 적용
+ 구현 간단, 단어 수보다 훨씬 정확
- 영어는 음절 세기가 복잡 (syllable counting)

문장 경계 동기화
UX 디자이너 관점
세밀한 청크 단위 대신 문장(sentence) 단위로만 동기화. 문장 경계는 tts/display 양쪽에서 동일 → 싱크 깨질 여지 없음. 각 문장 내부는 한 자막으로 표시 (2줄 제한 유지).
문장 분리 → silence에서 경계 탐지
→ 문장당 1-2개 자막
→ 내부 타이밍 = 문장 오디오 길이
+ 싱크 오차 최소, 로직 단순
- 긴 문장은 자막이 오래 유지됨

TTS 입력에 SSML <mark> 태그를 문장/청크 경계마다 삽입. TTS 엔진이 각 마크의 정확한 오디오 위치를 반환. 이를 자막 타이밍으로 직접 사용.
tts_script에 <mark> 삽입
→ TTS API (SSML 모드)
→ mark 타이밍 수신 → 자막 적용

데이터 사이언티스트 관점
display_script의 단어 수 대신, 숫자 패턴($XX,XXX, X.XX%)에 가중치를 부여. "$71,793"은 1단어지만 가중치 8 (발음 음절 수 기반). 일반 단어는 가중치 1. 이로써 비례 배분 정확도 향상.
display_script 청크별
→ 숫자 패턴 감지 → 가중치 부여
→ 가중 비례 배분 + silence snap
+ 기존 로직 소폭 수정, 코드 변경 최소

오디오 분석
에너지 기반 VAD 정렬
오디오 엔지니어 관점
현재 silence detection을 Voice Activity Detection(VAD)으로 교체. 음성 에너지 곡선에서 발화 구간을 정밀 추출. 각 발화 구간의 시작/끝을 자막 청크에 순차 매핑.
audio → VAD → 발화 구간 리스트
→ 청크 수 = 발화 구간 수로 조정
→ 1:1 매핑
+ silence보다 정밀한 구간 탐지
- VAD 라이브러리 추가 필요 (webrtcvad 등)
